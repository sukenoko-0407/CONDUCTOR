# CONDUCTOR_v1 Overall Overview

## 1. Purpose

CONDUCTOR_v1 connects two Claude Code Project Skills in a loose-coupled workflow for ligand-centric SAR grouping.

The two Skills are:

- `cs-conductor-desc-ligand`: converts compounds into descriptor and vector representations.
- `cs-conductor-grouping`: builds compound groups from structures, user-defined columns, and external vector representations.

The key v1 design decision is that the two Skills do not call each other directly. Instead, descriptor outputs are written under `descriptions/`, and Grouping reads any compatible CSV representation files from that directory.

```text
input CSV
  -> Desc Skill
       -> descriptions/<input_csv_stem>/*.csv

input CSV
  -> Grouping Skill
       -> validates input IDs and structures
       -> runs structure-based grouping
       -> reads descriptions/<input_csv_stem>/*.csv when present
       -> writes groups/<input_csv_stem>/*
```

This makes the connection extensible. Files in `descriptions/` may be generated by the Desc Skill or by other future descriptor-generation tools.

## 2. Scope

CONDUCTOR_v1 covers:

- RDKit ligand-only descriptor and fingerprint generation.
- Descriptor output organization under `descriptions/`.
- Group artifact generation under `groups/`.
- MCS grouping.
- Murcko scaffold grouping.
- BRICS/RECAP fragment grouping.
- Human/user-defined grouping columns.
- Descriptor-file-based clustering.
- Multiple clustering methods and multiple parameter patterns.
- Strict input ID validation.
- Final group filtering with a default minimum of 5 compounds per accepted group.

CONDUCTOR_v1 does not cover:

- QSAR model training.
- Activity cliff analysis.
- MMP analysis.
- Docking, ProLIF, PLIP, or pose analysis.
- SAR mechanism interpretation.
- New compound proposal.
- HTML report generation.
- Broad data preprocessing beyond what grouping requires.

## 3. Skill Responsibilities

### 3.1 Desc Skill

The Desc Skill converts compounds into descriptor CSV files.

Input:

- User-provided CSV containing a compound ID column and a SMILES column.

Output:

- `descriptions/<input_csv_stem>/Lxx_*.csv`
- `descriptions/<input_csv_stem>/errors.csv`
- `descriptions/<input_csv_stem>/run_metadata.json`

The Desc Skill should keep its responsibility narrow:

- infer or accept ID and SMILES columns
- parse SMILES
- calculate descriptor sets
- write descriptor CSVs
- record invalid molecule or descriptor errors

It should not generate grouping artifacts.

### 3.2 Grouping Skill

The Grouping Skill builds group artifacts.

Input:

- The original user CSV.
- Optional config JSON.
- Optional descriptor CSV files in `descriptions/<input_csv_stem>/`.

Output:

- `groups/<input_csv_stem>/compounds_master.csv`
- `groups/<input_csv_stem>/excluded_compounds.csv`
- `groups/<input_csv_stem>/group_registry.json`
- `groups/<input_csv_stem>/group_membership.csv`
- `groups/<input_csv_stem>/group_membership_matrix.csv`
- `groups/<input_csv_stem>/group_relations.json`
- `groups/<input_csv_stem>/selected_groups.json`
- `groups/<input_csv_stem>/group_summary.json`
- `groups/<input_csv_stem>/group_graph_packet.json`
- `groups/<input_csv_stem>/group_filter_report.json`
- `groups/<input_csv_stem>/grouping_warnings.json`
- `groups/<input_csv_stem>/grouping_manifest.json`
- diagnostic CSVs and figures when enabled

The main consolidated downstream artifact is `group_membership_matrix.csv`. It has one row per compound and one column per individual group. The first column is `compound_id`. Each group column is named by `group_id` and contains `0` or `1`, where `0` means false/non-member and `1` means true/member. A compound may belong to many groups, so many group columns may contain `1` in the same row.

`group_registry.json` maps each group ID to its label, type, source, and definition. `group_relations.json` and `group_graph_packet.json` are auxiliary lightweight relationship artifacts; they do not need to duplicate the full compound-by-group matrix.

The Grouping Skill should:

- detect or accept ID and SMILES columns
- stop with an error when input IDs are missing or duplicated
- retain only minimal structure preparation needed by grouping builders
- record invalid SMILES in `excluded_compounds.csv`
- run all configured grouping builders
- remove groups with fewer than the configured minimum compound count
- use descriptor CSVs from `descriptions/<input_csv_stem>/` when present

## 4. Directory Contract

### 4.1 Descriptor Outputs

Descriptor outputs must be grouped by input CSV stem.

```text
descriptions/
  chemble_jak2/
    L01_rdkit_0d_1d_2d.csv
    L02_ecfp4_bit.csv
    L03_ecfp4_count.csv
    ...
    errors.csv
    run_metadata.json
```

The subdirectory name is the input CSV filename without extension. For example:

```text
chemble_jak2.csv -> descriptions/chemble_jak2/
```

### 4.2 Group Outputs

Grouping outputs must also be grouped by input CSV stem.

```text
groups/
  chemble_jak2/
    compounds_master.csv
    excluded_compounds.csv
    group_registry.json
    group_membership.csv
    group_membership_matrix.csv
    group_relations.json
    selected_groups.json
    group_summary.json
    group_graph_packet.json
    group_filter_report.json
    grouping_manifest.json
```

The default Grouping output directory is:

```text
groups/<input_csv_stem>/
```

## 5. Descriptor CSV Contract

Grouping reads only CSV files from:

```text
descriptions/<input_csv_stem>/*.csv
```

Each descriptor CSV must contain:

- one compound ID column
- numeric feature columns

Known non-feature columns are ignored automatically, including:

- `compound_id`
- `canonical_smiles`
- `input_smiles`
- `original_smiles`
- `mol_parse_ok`
- `descriptor_error`
- `mol_error`
- `source_row_index`

All remaining usable numeric columns are treated as vector dimensions.

If multiple descriptor CSV files exist, Grouping clusters each file independently. It does not merge descriptor files by default.

## 6. Grouping Methods

CONDUCTOR_v1 uses several grouping families.

Structure-based grouping:

- human/user-defined groups
- Murcko scaffold groups
- MCS groups
- BRICS fragment groups
- RECAP fragment groups

Vector-based clustering:

- Butina
- hierarchical agglomerative clustering
- DBSCAN
- Louvain
- Leiden
- connected components

By default, all vector clustering methods run. Users can restrict methods by editing the Grouping config.

## 7. Parameter Sweep Policy

For vector-based clustering, a single parameter setting is not enough. CONDUCTOR_v1 therefore supports multiple parameter patterns per method.

Examples:

- Butina: multiple similarity thresholds
- hierarchical: multiple distance thresholds and linkage settings
- DBSCAN: multiple similarity thresholds and `min_samples`
- Louvain: multiple graph construction settings and resolution values
- Leiden: multiple graph construction settings and resolution values
- connected components: multiple similarity thresholds

Each method and parameter pattern produces separate groups and diagnostic records.

## 8. Input Validation Policy

The original input CSV must have valid compound IDs.

Grouping must stop before artifact generation when:

- the ID column is missing
- any compound ID is blank
- any compound ID is duplicated

These cases are treated as hard input errors. Grouping should notify the user instead of silently generating replacement IDs or renaming duplicates.

Invalid SMILES are not hard input errors. They are recorded in `excluded_compounds.csv` and excluded from structure-dependent grouping builders.

## 9. Preprocessing Policy

Grouping should not perform broad preprocessing.

Allowed minimal preparation:

- ID validation
- SMILES parse check
- canonical SMILES creation for valid structures
- invalid SMILES recording
- optional Wet/Virtual normalization only when needed by MCS sampling

Excluded preprocessing:

- activity normalization
- pIC50 or IC50 conversion
- duplicate ID rewriting
- missing ID generation
- descriptor scaling beyond clustering-local numeric preparation
- imputation policy beyond descriptor-file-local handling
- SAR interpretation

If broader preprocessing becomes necessary, it should be implemented as a separate preprocessing Skill.

## 10. Main Implementation Documents

The implementation specification is:

## 11. v1 Verification Snapshot

The current v1 implementation was tested with `chemble_jak2.csv`.

Observed results:

- input compounds: 231
- invalid SMILES: 0
- Desc outputs: L01-L11 under `descriptions/chemble_jak2/`
- accepted groups after `min_compound_count = 5`: 2601
- dropped groups by final group filter: 2372
- `group_membership_matrix.csv`: 231 rows x 2602 columns
- matrix values: `0` and `1` only
- `group_relations.json`: capped by `relations.max_relations = 5000`

Runtime warnings were expected:

- no Wet/Virtual column was found, so all compounds were treated as Wet
- MCS pair mining was capped at `max_mcs_pair_count = 1000`
- final group filtering removed groups with `compound_count < 5`
